# SyncFuzz proof

SyncFuzz is a **working name** for a local falsification harness that asks one
question: after several real browser clients act under adverse schedules, do
their application-defined canonical states converge and still match the
semantic model?

This repository is intentionally smaller than a product. It launches isolated
Chromium contexts with Playwright, drives semantic actions through a six-method
adapter, generates schedules from a seed with fast-check, minimizes failures by
replaying candidate schedules, and emits a short JSON + Markdown counterexample.

The included targets are synthetic micro-apps. They make the harness testable;
they do not prove integration cost or fault coverage on production systems.

## Five-second demo

```sh
pnpm install
pnpm exec playwright install chromium
pnpm demo
```

The demo plants a disclosed WebSocket message-loss mutant. Its generated replay
looks like this:

```text
Seed 10000, replay path ddmin/3:5,0:2
5 steps -> 1 step
1. client 1 increments by 1
FAIL semantic-model: canonical value differs from accepted semantic total 1
```

Artifacts land under `output/playwright/`. The checked-in example is
[`examples/websocket-drop-first-outbound.md`](examples/websocket-drop-first-outbound.md).

## What is real

- Two to five clients are separate Playwright `BrowserContext`s, not simulated
  objects or tabs sharing storage.
- WebSocket, Server-Sent Events + HTTP, and REST polling exercise three distinct
  inbound transport shapes. A fourth polling + localStorage fixture measures
  adapter integration.
- Offline and reconnect use `BrowserContext.setOffline`; reload uses
  `Page.reload`; client death closes the page and relaunch creates a new page in
  the same context.
- The runner observes the browser app through semantic actions, canonical
  snapshots, application-defined quiescence, and invariants.
- Schedules and action values are generated by fast-check from an integer seed.
  Replay-driven delta debugging is used for shrinking because browser timing can
  invalidate purely structural shrinks.

## What is synthetic

- The 20 fault-gate cases are explicit implementation mutants, not naturally
  discovered production bugs.
- Delay, single-message drop, and clock skew are fixture-level transport/lifecycle
  controls. They are not claims of arbitrary TCP packet control.
- The fixtures use a local in-process server, no authentication, and no service
  worker. Remote queues and third-party infrastructure are outside this proof.

## Commands

```sh
# Type checks and tests
pnpm check
pnpm test
pnpm test:browser

# One minimized counterexample
pnpm demo

# All 20 disclosed mutants
pnpm syncfuzz fault-suite

# Baseline resilience gap not covered by the fixture's happy-path test
pnpm syncfuzz novel

# Independent PR-size timing run
pnpm campaign

# Re-run a saved failure exactly
pnpm syncfuzz replay output/playwright/faults/websocket-drop-first-outbound.json

# Recompute the gate report locally
pnpm proof
```

Node 22 or newer and pnpm are required. The browser install is local Playwright
state; no account or external service is needed after dependency installation.

## Adapter contract

An application adapter supplies:

```ts
interface SyncAdapter {
  reset(options): Promise<void>;
  launch(clientId, options): Promise<ClientHandle>;
  relaunch(client, options): Promise<void>;
  act(client, semanticAction): Promise<void>;
  canonicalSnapshot(client): Promise<CanonicalSnapshot | null>;
  waitForQuiescence(clients, timeoutMs?): Promise<void>;
  assertInvariants(clients): Promise<InvariantFailure[]>;
  control(client, faultOrLifecycleStep, options): Promise<void>;
  close(clients): Promise<void>;
}
```

The proof keeps reset/launch, action, observation, quiescence, invariant, and
fault responsibilities explicit. See [`docs/adapter-guide.md`](docs/adapter-guide.md)
and [`docs/architecture.md`](docs/architecture.md).

## Gate discipline

`pnpm proof` writes `outputs/syncfuzz-proof-report.{json,md}` with an exact status
for every technical pass-or-kill gate. A green report on these local fixtures is
not a production validation. In particular, the fourth-app time measures a
purpose-built target against an already-known contract, and the novel-failure
case is a real baseline resilience gap in a synthetic fixture.

## Repository map

```text
src/                 runner, adapter contract, CLI, generation, shrink, replay
fixtures/            local browser apps and fixture server
tests/               core, existing-test baseline, and fourth-app integration
examples/            checked-in minimal replay
evidence/            integration timing record
docs/                architecture, adapter, metrics, and limitations
outputs/             final local proof report (generated)
output/playwright/   ephemeral browser replay artifacts (ignored)
```
